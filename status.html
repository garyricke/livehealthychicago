Open Direction / Future Decisions · 5 items
  • Portal & Celebrate the Check — live; access is the only open question
    Team
    Rachel's 6/17 ask was to "activate" these — but both are already technically live and reachable now: the resource portal at /portal (short link → portal.html) and Celebrate the Check at /check (with a working QR). So there's no go/no-go decision left — the only open question is whether we want to change access in some way (who can reach them, gating, or how we promote them). The segment-by-user-type idea was handled with audience tags; analysis below.
    Analysis — segment by user type (recommend: don't, yet). The instinct is reasonable, but on balance the evidence supports not adding a user-type layer right now. Three points:
    • The portal already segments by role. Two of its three tabs are user types — Clinical / Technical (clinicians, nurses, admins) and CHW Resources (community health workers); the third, Social Media Kit, is content-type. So role-based browsing already exists. A separate user-type axis would largely duplicate the category axis.
    • A second axis hides resources. Adding user type on top of category creates a matrix (type × category). A given resource — say a BP-measurement guide useful to clinicians, CHWs, and health educators — then has to be either duplicated (maintenance drift) or filed under one role and hidden from the others. That's the exact "good resources get buried" risk, and it's real. Many resources here are genuinely cross-role.
    • The library is small and there's no per-user identity. ~18 resources across 3 tabs (~6 each) — a user scans a whole tab in seconds, so faceted segmentation solves a scale problem the portal doesn't have. And true per-user segmentation needs per-user logins/roles; today there's a single shared viewer password, so "who is a clinician vs. educator?" isn't even knowable without building an identity layer first.
    Cheaper path that captures the benefit: if a role lens is wanted, sharpen the existing tabs (role-forward labels like "For Clinicians / For CHWs / For Health Educators / Promotion") and/or add optional, multi-select role tags with a "show all" default filter — tags narrow without hiding, a resource can carry several, and nothing gets buried. Reserve a hard role-gated structure for when (a) the library outgrows easy browsing and (b) there's a real need for role-private content (resources a role shouldn't see) — a different requirement from findability. Net: agree with the pushback; revisit only on those triggers.
    Update (6/23): audience tags shipped ✓. Built the lightweight tag path above — 6 audience tags, multi-select, with a default-"All" filter bar that narrows without hiding. Live in the portal now; see the "Audience Tags Added" note in today's timeline entry. A hard role-gated structure remains deferred to the two triggers above.
